Loose & perishable items
in the Thing system

This document proposes how to extend Shwapno's existing Thing-based WMS to handle loose commodities (rice, oil) and perishable items (Rui fish, tomato, meat) — without breaking the current CG item architecture.

The problem

Why loose items break the current system

Today, 1 Coca-Cola can = 1 Thing row. The system creates N rows for N units. This works perfectly for discrete CG items because each unit has a fixed, indivisible size.

For loose items like Rui fish (sold by kg, cut per customer request) or rice (any decimal quantity), there is no fixed atomic unit. A customer can ask for 3.25 kg or 0.875 kg. A fish can be cut into heads, fillets, tails. Creating one row per gram is impractical. Creating one row per kg causes edge cases with every decimal sale.

The solution
One batch = one Thing. Every movement = one Event.

When 99.5 kg of Rui fish arrives from a supplier, the system creates exactly one Thing row (the batch identity). Every subsequent operation — packing, transfer, transit loss, sale, wastage, transmutation — appends an Event row to the log. Current stock at any location is always derived by summing events. The Thing row itself is never modified after creation (except state).

Core rule: A new Thing is created ONLY when goods arrive from an external supplier (Purchase Order). Internal movements — DC to outlet, outlet to outlet — never create new Things. They only create events on the existing Thing.

When a Thing row is created

Only at GRN — when goods arrive from a supplier PO. Each unique batch arrival = one Thing. For transmutation outputs, new Things are born (one per output SKU).

When an Event row is created

For every movement: DC packing, outlet receiving, transit loss, customer sale, wastage, adjustment, return, transmutation in/out. One operation = one event row.

Transmutation

Whole fish → heads + fillets + tails. The original Thing gets a TRANSMUTATION_OUT event. New Things are born for each output SKU. A TX record bridges input to outputs.

What never changes

The Thing row's original_qty is immutable. Stock is always calculated, never stored. Corrections are new ADJUSTMENT events — no row is ever edited or deleted.

UNCHANGED

The single most important mental shift

For CG items, the Thing is the stock unit. For loose items, the Thing is the identity of a batch — the events describe the stock. Both live in the same table. Both use the same state machine. The only structural difference is how quantity is handled: counted vs calculated.

-- Primary lookup: all events for a Thing CREATE INDEX idx_tqe_thing_id ON thing_quantity_events(thing_id); -- Time-range queries and partition pruning CREATE INDEX idx_tqe_happened_at ON thing_quantity_events(happened_at); -- Stock at a specific location CREATE INDEX idx_tqe_location ON thing_quantity_events(location_to, location_from); -- FEFO queries — order by earliest expiry CREATE INDEX idx_tqe_exp_date ON thing_quantity_events(exp_date); -- Ref lookups (e.g. all events for a transfer order) CREATE INDEX idx_tqe_ref ON thing_quantity_events(ref_type, ref_id);
Core stock query
-- Current stock of product at all locations (with FEFO ordering) SELECT t.product_id, e.exp_date, COALESCE(e.location_to, e.location_from) AS location, SUM(CASE WHEN e.direction = 'IN' THEN e.qty ELSE 0 END) - SUM(CASE WHEN e.direction = 'OUT' THEN e.qty ELSE 0 END) AS current_qty FROM thing_quantity_events e JOIN things t ON t.id = e.thing_id WHERE t.product_id = '<product-uuid>' GROUP BY t.product_id, e.exp_date, location HAVING SUM(CASE WHEN e.direction = 'IN' THEN e.qty ELSE 0 END) - SUM(CASE WHEN e.direction = 'OUT' THEN e.qty ELSE 0 END) > 0 ORDER BY e.exp_date ASC; -- natural FEFO
Immutability trigger: Add a PostgreSQL trigger on the things table that raises an exception if any UPDATE attempts to change original_qty, unit, or product_id on a row where is_loose = true. These fields are write-once.
Edge cases

Every known edge case — and how the model handles it

These are the situations that break naive implementations. Each one is handled explicitly.

PO qty vs received qty mismatch

PO raised for 100 kg. Supplier delivers 99.5 kg.

Handling: Thing 101 is created with original_qty = 99.5 (what physically arrived). The 0.5 kg shortfall is handled at PO reconciliation level — not in Things. Things only ever record physical reality.

Thing created at actual qty PO discrepancy handled separately

Internal transfer — outlet receiving never creates a new Thing

DC packs 60 kg and sends to Outlet 1. Outlet 1 receives 58 kg.

Handling: No new Thing. Two events on Thing 101: RECEIVED_IN (58 kg IN) and TRANSIT_LOSS (2 kg OUT). The original Thing retains full traceability back to the supplier PO. Creating a new Thing here would break the chain.

RECEIVED_IN event TRANSIT_LOSS event

Arbitrary decimal customer qty

Customer asks for 3.25 kg, or 0.875 kg, or 1.1 kg.

Handling: A SOLD event with qty = 3.25. The NUMERIC(12,4) type supports up to 4 decimal places. Stock is recalculated as SUM(IN) − SUM(OUT). No rounding or approximation needed.

SOLD event with exact decimal

DC remainder — stock stays without an event

99.5 kg received. 60 + 37 + 2 packed. That leaves 0.5 kg at DC.

Handling: No event needed. DC stock = original_qty minus all OUT events at DC. The 0.5 kg is always implied. The system never needs to explicitly "mark" remaining stock — it's derived from the math.

No event — derived from math

Partial transmutation

52.45 kg of whole fish available. Butcher takes only 20 kg to process. The rest continues selling.

Handling: TRANSMUTATION_OUT event for 20 kg on Thing 101. New Things 102, 103, 104 are born. Thing 101 still has 32.45 kg untouched and continues accepting SOLD events normally. Two SKUs coexist simultaneously.

TRANSMUTATION_OUT on Thing 101 New Things 102, 103, 104 born

Processing loss during transmutation

20 kg whole fish goes in. Only 17.5 kg comes out as new products. 2.5 kg is bones and offcuts.

Handling: WASTAGE event of 2.5 kg OUT on Thing 101 (linked to TX-001). Balance: 20 = 17.5 + 2.5. Every gram accounted for. The processing loss is attributed to the original Thing, not the new outputs.

WASTAGE event on source Thing Balance always closes

Wrong quantity entered — correction

Staff entered 60 kg dispatched but it was actually 58 kg.

Handling: Never edit the existing event. Append an ADJUSTMENT event for +2 kg IN with reason "receiving correction, ref E-002". The event log is append-only forever. Full audit trail preserved. Any query after the adjustment returns the correct stock.

ADJUSTMENT event appended Original event untouched

Sub-lot expiry — same batch, different expiry dates

100 kg rice arrives. 60 kg expires June 1, 40 kg expires June 15 (different sub-lots from same supplier shipment).

Handling: One Thing 101. Two RECEIVED events — one with exp_date = June 1 for 60 kg, one with exp_date = June 15 for 40 kg. FEFO queries order by exp_date ASC and pick the earliest-expiring stock first.

Two RECEIVED events, one Thing FEFO handled naturally

Multiple outlets, one Thing

Thing 101 has stock at DC (0.5 kg), Outlet 1 (52.45 kg), and Outlet 2 (36.5 kg) simultaneously.

Handling: Stock per location is derived by filtering events by location_to or location_from. The snapshot table pre-computes this breakdown every 15 minutes for dashboards. Real-time stock queries hit the event log directly.

Same Thing, filtered by location Snapshot for dashboard performance

Customer return

Customer returns 1.5 kg of rice bought earlier.

Handling: RETURN event on the original Thing — 1.5 kg IN at the outlet. The qty re-enters the system cleanly. If the item is unsaleable (damaged by customer), follow the RETURN with a WASTAGE event.

RETURN event (IN) Optional WASTAGE if unsaleable
Scalability & performance

Making it fast as Shwapno grows

The event model scales well by design — it is append-only and naturally partitionable. Here is what to do and when.

~500
events / day (conservative estimate)
50 loose SKUs × DC + 25 outlets, each with 2–4 movements per day. At this volume, no partitioning needed. Plain indexes are sufficient.
~180K
events / year
Postgres handles tens of millions of rows without partitioning. You have several years of growth before the table needs range partitioning.
10M+
rows — trigger range partitioning
At this point, partition thing_quantity_events by happened_at monthly. Queries that filter by date automatically use only the relevant partition.
15 min
snapshot rebuild interval
The thing_stock_snapshot table is rebuilt every 15 minutes via pg_cron. Dashboards and reports hit the snapshot. Only live transactions hit the raw event log.
Implementation checklist — ordered by priority
  • 1
    Add columns to things table (is_loose, original_qty, unit, parent_tx_id)
    One migration, fully additive. No existing rows affected. CG item behaviour unchanged. Ship this first so the table is ready.
  • 2
    Create thing_quantity_events table + all indexes
    This is the foundation of everything. All five indexes should be created on launch — not added later. Include the immutability trigger on things.
  • 3
    Build receiving flow for loose items (GRN)
    User enters batch weight at receiving dock. System creates 1 Thing + 1 RECEIVED event. This unblocks operations immediately.
  • 4
    Build FEFO stock query and transfer packing logic
    Before packing a transfer, the app queries stock by exp_date ASC and picks the earliest-expiring batch first. Critical for perishables.
  • 5
    Build transfer confirmation + transit loss recording
    When outlet confirms receipt, app writes RECEIVED_IN and TRANSIT_LOSS events. Transfer order receives actual_received_qty for reconciliation.
  • 6
    Build sale event flow at outlet POS
    Customer weighs item, POS writes SOLD event. Any decimal qty supported. Outlet stock recalculated on next query.
  • 7
    Build wastage and adjustment event flows
    Wastage logged by store manager. Adjustments logged by supervisor with mandatory reason text. Append-only — no edits.
  • 8
    Create thing_stock_snapshot + pg_cron rebuild job
    Simple materialized view pattern. Rebuild every 15 minutes. All dashboard and reporting queries point here, not at the event log directly.
  • 9
    Build transmutation flow + transmutation tables
    Most complex feature. Requires UI for butcher to log outputs and weights. Create transmutations, transmutation_inputs, transmutation_outputs tables. Ship last — can be manual-entry initially.
  • 10
    Add range partitioning on thing_quantity_events by happened_at
    Not needed at launch. Add when table exceeds 5–10M rows. pg_partman can automate monthly partition creation.
Query performance rules

Live stock queries

Always filter by thing_id or product_id + location. Never do a full table scan. The idx_tqe_thing_id index makes per-batch queries fast even at millions of rows.

Dashboard / reports

Always hit thing_stock_snapshot — never the raw event table. The snapshot is rebuilt every 15 min. For end-of-day reports, rebuild on-demand before the report runs.

FEFO picking

Order by exp_date ASC in the stock query. The idx_tqe_exp_date index makes this fast. Always use the live event table (not snapshot) for picking to avoid stale data.

Audit / traceability

Queries that trace a transmutation chain (what did TX-001 produce? where did Thing 102 come from?) are rare and can be slow. No index optimisation needed — these are investigative queries.

Architect's note: The event log is append-only by design. This means you can safely add read replicas and point all dashboard/report traffic at replicas, keeping the primary Postgres instance free for live transaction writes only. This is the single highest-impact performance improvement at scale.
